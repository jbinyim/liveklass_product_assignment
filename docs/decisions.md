# 설계 결정 기록 (Decision Log)

기획·구현 과정에서 마주친 애매한 지점, 트레이드오프가 있는 선택, 명세가 비워둔 결정을 기록한다.
README 「요구사항 해석 및 가정」 / 「설계 결정과 이유」 작성 시 이 문서를 1차 소스로 사용한다.

## 기록 형식

각 결정은 아래 항목으로 구성한다.

- **ID & 제목** — `D001. 결정 제목` (순번 부여)
- **상태** — `결정됨` / `재검토 필요` / `폐기됨`
- **명세 근거** — 명세 어느 부분에서 비롯된 결정인지 (없으면 "명세 외 자체 판단"이라고 명시)
- **검토한 옵션** — A/B/C... 각각 장단점
- **결정** — 어느 옵션, 변형 여부
- **이유** — 왜 그 옵션을 골랐는지, 다른 옵션을 버린 이유

작성 원칙:
- 옵션은 최소 2개 이상 (대안 검토 흔적 남기기)
- "명세 직접 인용" 부분은 그대로 옮겨 적기
- 결정이 뒤집히면 기존 항목을 지우지 말고 상태를 `폐기됨`으로 변경 + 새 ID로 추가

## D001. 개인↔단체 전환 시 데이터 처리 정책

**상태:** 결정됨

**명세 근거:**
- 평가 기준 1) "특히 봐야 할 애매한 지점들": *"단체 신청에서 개인 신청으로 전환 시 단체 관련 데이터를 어떻게 처리하는가"*
- 같은 표 「조건부 필드 전환」 높은 숙련도: *"타입 변경 시 관련 데이터 초기화 + 확인 대화상자"*

**검토한 옵션:**

- **A. 즉시 폐기 + 항상 확인 대화상자** — 전환 시 무조건 모달, 확인하면 반대편 데이터 삭제
  - 장점: 명확한 정책, 평가 기준 표현 그대로 충족
  - 단점: 빈 상태에서도 모달 떠서 잡음
- **A 변형. 유효 입력이 있을 때만 확인 모달** — 반대편 필드가 비어 있으면 즉시 전환, 입력값이 있을 때만 모달
  - 장점: A의 평가 기준 충족 + 빈 상태 잡음 제거
  - 단점: 분기 로직이 약간 늘어남
- **B. 백그라운드 보관 + 재전환 시 복원** — 전환해도 데이터 유지, 제출 페이로드에서만 제외
  - 장점: 실수 복구 쉬움
  - 단점: discriminated union 정합성과 충돌, 복원 동작이 사용자에게 보이지 않으면 혼란
- **C. 즉시 폐기, 모달 없음** — 가장 단순, UX 위험

**결정:** A 변형

**이유:**
- 명세 표의 "초기화 + 확인 대화상자" 기준을 그대로 충족
- 빈 상태에서 무의미한 모달이 뜨지 않아 일반 흐름의 마찰 최소
- B는 "데이터 정합성" 평가 항목과 어긋나고 README에 설명할 정책이 모호해짐
- C는 평가 기준의 "낮은 숙련도" 예시(데이터 방치)에 가까움

## D002. 신청자(applicant)와 참가자(participants) 역할 해석

**상태:** 결정됨

**명세 근거:**
- API 스키마 `GroupEnrollmentRequest`가 `applicant`와 `group.participants`를 별도 필드로 분리
- 평가 기준 1) "특히 봐야 할 애매한 지점들": *"참가자 명단의 이메일 중복을 어떻게 처리하는가"*

**검토한 옵션:**

- **해석 1. 신청자 ≠ 참가자 (역할 분리)** — 신청자는 행정 주체(폼 작성·연락), 참가자는 수강 주체. 동일인일 경우 신청자에는 본인 정보, 참가자에는 별도 인원
  - 장점: API 모델의 분리 의도와 일치, headCount 의미 명확, 검증 로직 단순
  - 단점: 신청자 본인이 듣고 싶을 때 이메일을 별도로 마련해야 함
- **해석 2. 신청자 ∈ 참가자 가능 (중복 허용)** — HR팀장이 본인도 듣는 케이스 인정
  - 장점: 자연스러운 케이스 수용
  - 단점: headCount = 5인데 신청자 + 참가자 5명 = 6명 듣는지 5명 듣는지 모호. 정원 검증이 꼬임

**결정:** 해석 1 (역할 분리)

**이유:**
- API 모델이 `applicant`와 `participants`를 별도 필드로 둔 시점에서 역할 분리가 의도된 설계로 읽힘
- headCount = participants.length 등식이 깨지지 않아 정원 로직 단순
- README 「요구사항 해석 및 가정」에 "신청자는 행정 주체, 참가자는 수강 주체" 한 줄로 명확히 설명 가능

## D003. 참가자 명단 이메일 중복 검증 범위

**상태:** 결정됨

**명세 근거:**
- 평가 기준 1) "특히 봐야 할 애매한 지점들": *"참가자 명단의 이메일 중복을 어떻게 처리하는가"*
- 같은 표 「유효성 검증 시점」 높은 숙련도: *"필드 blur 시 개별 검증"*
- D002에서 신청자 ≠ 참가자로 결정한 결과를 따름

**검토한 옵션 (검증 범위):**

- **A. 참가자 간 중복 + 신청자 vs 참가자 중복 차단** — 담당자(contactPerson)는 단순 문자열이라 검증 대상 외
- **B. 참가자 간 중복만 차단** — 신청자/참가자 동일 허용 → D002와 충돌
- **C. 검증 안 함, 서버 위임** — 명세의 "애매한 지점" 회피, 평가 손해

**검토한 옵션 (검증 시점):**

- **(i) onChange 즉시** — 입력 중 빨간 줄 떴다 사라졌다 산만
- **(ii) blur 시점** — 명세 직접 인용
- **(iii) 스텝 전환 시에만** — 명세 「스텝 전환 시 해당 스텝 검증 + blur 시 개별 검증」의 후자 결여

**결정:** A안 + (ii) blur 시점

**이유:**
- D002에서 신청자 ≠ 참가자로 결정한 이상, 같은 이메일은 데이터 모순
- 담당자(contactPerson)는 명세상 이메일 필드가 없는 단순 문자열이라 검증 대상 자체가 아님
- blur 시점은 명세가 "높은 숙련도" 기준으로 직접 명시
- zod `refine`으로 schema 레벨에 위치시켜 UI와 분리 (평가 기준 2번 「유효성 검증 로직이 UI와 분리되어 있는가」 충족)

## D004. 정원 거의 찬 강의 UX

**상태:** 결정됨

**명세 근거:**
- 평가 기준 1) "특히 봐야 할 애매한 지점들": *"정원이 거의 찬 강의를 선택했을 때 UX를 어떻게 처리하는가"*
- `Course` 타입에 `maxCapacity`, `currentEnrollment` 필드 존재
- 면접 연계 질문 4번: *"유효성 검증을 클라이언트와 서버 양쪽에서 해야 하는 이유는 무엇인가요?"*

**검토한 옵션 (잔여석 표시 — D004-a):**

- **A. 항상 잔여석 숫자 표시** — "잔여 12석 / 50석"
- **B. 임계 이하일 때만 강조** — 평소 숨김, 잔여 5석 이하 빨간 배지
- **C. 비율 / 정성 표현만**

**검토한 옵션 (정원 가득 찬 강의 — D004-b):**

- **A. 목록에서 숨김** — 컨텍스트 손실
- **B. 회색 처리 + "마감" 배지 + 클릭 차단** — 존재 노출, 선택 불가
- **C. 클릭 가능, 선택 후 서버 에러** — 끝까지 입력하고 거부당함

**검토한 옵션 (단체 인원수 > 잔여석 — D004-c):**

- **A. 클라이언트 검증만** — 즉시 피드백, 동시성 위험
- **B. 서버 `COURSE_FULL` 응답에만 의존** — 동시성 안전, 마지막에야 거부 → UX 나쁨
- **C. 클라이언트 + 서버 양쪽** — 즉시 피드백 + 동시성 권위

**결정:**
- D004-a: B (잔여 5석 이하부터 "마감임박" 배지)
- D004-b: B (회색 + 마감 배지 + 클릭 차단)
- D004-c: C (클라이언트 즉시 검증 + 서버 최종 검증)

**이유:**
- D004-a: 평소 시야 깔끔, 임박 시점에만 시각적 압박. 임계값 5석은 휴리스틱 (필요 시 조정).
- D004-b: 컨텍스트 유지하면서 명확한 차단. A는 사용자가 "내가 보던 강의가 사라졌다"고 혼란.
- D004-c: 면접 질문 4번이 정확히 이 분리를 묻고 있음. 클라이언트(UX) + 서버(정합성)는 책임이 다름. 둘 다 구현해야 답변 거리가 됨.

## D005. 에러 시나리오별 복구 UX

**상태:** 결정됨

**명세 근거:**
- API 스키마 `ErrorResponse` 및 주요 에러 코드: `COURSE_FULL` / `DUPLICATE_ENROLLMENT` / `INVALID_INPUT`
- 평가 기준 3) 「서버 에러 코드 처리」: *"COURSE_FULL, DUPLICATE_ENROLLMENT 등을 사용자에게 의미 있게 전달하는가"*
- 면접 연계 질문 3번: *"네트워크 에러와 비즈니스 에러를 구분하나요?"*
- 평가 기준 1) 표 「에러 표시」 높은 숙련도: *"에러 필드로 포커스 이동, 시각적 강조, 스크롤 처리"*

**검토한 옵션 및 결정:**

### D005-a. `COURSE_FULL` 처리

- A. 자동으로 1단계로 이동 (토스트만)
- B. 모달 + 사용자 선택 ("강의 선택으로" / "닫기") ✅
- C. 3단계 인라인 에러 + 이동 버튼

→ **결정: B.** 사용자가 흐름의 주도권을 쥠. 자동 이동은 입력값이 날아간 듯한 인상. 인라인 표시는 시각적으로 약함.

### D005-b. `DUPLICATE_ENROLLMENT` 처리

- A. 모달 안내 + 닫기만 (재시도 차단) ✅
- B. 모달 + 다른 강의 선택 버튼
- C. 인라인 에러 + 재시도 차단

→ **결정: A.** 재시도가 의미 없는 에러. 메시지만 명확히 전달하고 종료. 가짜 링크(마이페이지 등)는 만들지 않고, "고객센터 문의" 같은 일반 안내로 마무리.

### D005-c. `INVALID_INPUT` 처리

- A. 해당 스텝으로 자동 이동 + 첫 에러 필드 포커스 ✅
- B. 3단계 인라인 에러 + "수정하러 가기" 버튼
- C. 메시지만 띄우고 사용자가 직접 "수정" 링크로 이동

→ **결정: A.** `details`를 RHF `setError`로 필드별 매핑한 뒤, 첫 에러 필드의 스텝으로 이동 + 포커스 + 스크롤. 클라이언트 검증을 통과한 입력이 서버에서 거부되는 건 드문 케이스이므로 즉시성이 더 중요. 평가 기준의 "포커스 이동 + 스크롤 처리"를 만족.

### D005-d. 네트워크 에러 처리

- A. 토스트 + 수동 재시도 (모든 요청) — POST는 안전하지만 GET이 흔들리는 순간에도 사용자가 누름
- B. 자동 재시도 1~2회 후 실패 시 토스트 (모든 요청) — POST는 멱등성 위험
- **C. GET만 자동 재시도 1회, POST는 토스트 + 수동 재시도** ✅

→ **결정: C.**
- 강의 목록 GET: TanStack Query `retry: 1`
- 제출 POST: `retry: 0`, 실패 시 토스트로 재시도 유도
- 이유: POST는 멱등하지 않아 서버에 부분 처리되어 있을 수 있음. 자동 재시도는 중복 신청 위험. 면접 질문 3번(네트워크 vs 비즈니스 에러 구분)에 대해 "메서드별 재시도 정책 분리" 답변 거리 확보.

**비즈니스 에러와 네트워크 에러의 구분:**
- 비즈니스 에러(`COURSE_FULL`/`DUPLICATE_ENROLLMENT`/`INVALID_INPUT`): 서버가 정상 응답한 4xx, 사용자 행동/상태에 의미가 있음 → 시나리오별 맞춤 UX
- 네트워크 에러(fetch 실패, 5xx, 타임아웃): 서버 도달 실패 또는 일시적 장애, 사용자 행동과 무관 → 재시도 유도

## D006. 임시 저장(localStorage) 정책

**상태:** 결정됨

**명세 근거:**
- 구현 범위 「선택 구현 (추가 점수)」: *"임시 저장: 새로고침 후에도 입력 데이터 복구 (localStorage 등)"*

**검토한 옵션 및 결정:**

### D006-a. 키 스키마

- A. 단일 키 `enrollment-draft`
- B. 강의별 키 `enrollment-draft:{courseId}`
- **C. 단일 키 + 내부에 `courseId` 포함** ✅

→ **결정: C.** 키는 `livclass:enrollment-draft:v1` (네임스페이스 + 버전). 사용자는 보통 한 번에 한 강의를 신청하므로 B는 과도. C는 강의 변경 시 폐기 판단 가능.

### D006-b. 저장 시점

- A. 입력할 때마다 즉시
- **B. debounce 500ms 후 저장** ✅
- C. 스텝 전환 시에만

→ **결정: B.** RHF `watch` + debounce 500ms. 새로고침 시 직전 입력까지 보존.

### D006-c. 복원 동작

- A. 자동 복원, 알림 없음
- **B. "이전 입력값을 복구하시겠어요?" 모달 → 복원 / 폐기** ✅
- C. 작은 알림 배너

→ **결정: B.** 명시적 동의로 안전. 사용자가 의도하지 않은 복원 방지.

### D006-d. 폐기 시점

- 제출 **성공** 시 즉시 클리어
- 복원 모달에서 "폐기" 선택 시
- `courseId` 변경 시 — 이전 강의 입력은 무의미하므로 폐기
- 단체 ↔ 개인 전환 시 — 유지 (D001 모달이 별도로 처리)
- TTL: **24시간** 초과 시 자동 폐기

### D006-e. 저장 범위

- A. 모든 필드 저장
- B. 민감 필드(전화/이메일) 제외
- **C. 모든 필드 저장 + README에 한계 명시** ✅

## D010. Mock API 구현 방식 — MSW

**상태:** 결정됨

**명세 근거:**

  - 동작: Service Worker(브라우저) / Node 인터셉터(테스트)로 fetch를 가로채 가짜 응답 반환

    - 클라이언트 코드는 MSW 존재를 모름 → 진짜 백엔드 교체 시 변경점 0
    - dev/test에서 동일한 handlers 재사용 → 테스트 정책의 "MSW 통합 테스트" 무료
    - HTTP 레벨에서 가로채므로 fetch 인터페이스/에러 시뮬레이션이 진짜와 동일
    - 비즈니스 에러 시나리오를 status code + body로 자연스럽게 반환
  - 단점: 초기 셋업(Service Worker 등록, Vitest 셋업) 살짝 있음
- **B. Next.js Route Handler (`app/api/*`)** — Next.js 내장 API 라우트로 mock 작성
  - 장점: 별도 라이브러리 불필요, Next.js 친화적
  - 단점: dev에서만 동작 (Vitest 환경에서 별도 셋업 필요), mock과 진짜 백엔드의 경계가 모호해짐, 테스트에서 MSW 같은 격리 효과 없음

  - 장점: 가장 단순
  - 단점: fetch 인터페이스 우회 → 실제 네트워크 동작 검증 못 함, POST/에러 시뮬레이션 어색, 테스트에서 mock 격리 어려움

  - 장점: 진짜 서버처럼 동작
  - 단점: dev 시 별도 프로세스 필요, 제출 요건 *"`npm i && npm run dev`로 즉시 실행 가능"* 위반 위험, 테스트에서 또 띄워야 함

- 명세가 "MSW, JSON 파일, 로컬 서버 등 방식 자유"로 첫 번째 예시로 MSW를 명시 → 평가자에게 친숙

**결정:** B

| 레이어 | 역할 | 의존 | 예시 |
|---|---|---|---|
| page.tsx | 라우팅 진입, 컴포지션 | hooks, _components | `applicant/page.tsx` |
